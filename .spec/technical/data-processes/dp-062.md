## Description

Manages the notification → acceptance → activation lifecycle for randomly assigned governance roles (auditor, reviewer, oversight, review_body). Ensures a citizen actively accepts the assignment before the role becomes active, and re-runs selection if the citizen declines or times out.

Applies to: `governance_role.type` in (auditor, reviewer, oversight, review_body).
Does NOT apply to: citizen (auto-granted), expert (auto-granted on competency), operator (DP-063), protocol_council (DP-064).

### Flow

1. DP-040 writes a `civic_assignment` row (TBL-024) with `status=pending_acceptance` and a `governance_role` row (TBL-032) with `status=pending`.
2. DP-039 dispatches notification to citizen: assignment details, term dates, estimated time commitment, and a 48-hour acceptance window.
3. Citizen accepts via `POST /assignments/{id}/accept` → sync write (DP-023 variant):
   - `civic_assignment.status` → `accepted`
   - `governance_role.status` → `active`, `term_start` = now()
   - Emit to audit.append (DP-036).
   - Emit acceptance confirmation notification (DP-039).
4. Citizen declines via `POST /assignments/{id}/decline`:
   - `civic_assignment.status` → `declined`
   - `governance_role` row deleted.
   - Re-enqueue to civic.assign for DP-040 to select a replacement citizen.
   - Decline recorded in `participation_record` (does not penalize, but is tracked for workload balancing).
5. If no response within 48 hours:
   - DP-058 (stale escalation cron) detects `pending_acceptance` assignments past deadline.
   - Treated as decline: same re-enqueue path as step 4.
   - Citizen receives an "assignment expired" notification.

### COI check before activation

Before `governance_role.status` transitions to active, DP-033 (COI auto-exclusion) is triggered. If an active conflict of interest is detected for the assignment's domain: assignment is declined automatically, role is not activated, citizen is notified, and DP-040 re-selects.
