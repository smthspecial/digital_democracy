## Description

Handles session creation, token refresh, and logout.

### Login flow

1. Actor submits government-linked credential (handle + credential proof).
2. auth-service verifies credential against `citizen.legal_identity_hash` (SRV-001 read).
3. If `citizen.status = suspended | revoked` → reject with no detail leak.
4. If `citizen.status = pending` → reject; prompt to complete identity verification (DP-002).
5. Check enrolled MFA factors (TBL-038). If none enrolled → session created at T1 only; all T2 writes blocked until enrollment (DP-060).
6. If MFA factors exist → issue MFA challenge (DP-061). On success, session tier is set based on factor type used (T2 minimum; T3 if passkey or facial used).
7. Write `session` row (TBL-037): access token (15-min TTL) + refresh token, device fingerprint, IP subnet.
8. Write `auth_event` row (TBL-039, event_type=login_success or login_failure).
9. Emit to audit.append (DP-036).
10. Return signed access token to client. Refresh token is httpOnly cookie only — never in response body.

### Token refresh flow

1. Client presents refresh token cookie.
2. auth-service looks up `session` row by refresh_token_hash.
3. Verify device fingerprint and IP subnet match. Mismatch → trigger DP-066 (anomaly), suspend session, return 401.
4. Rotate refresh token (old hash invalidated, new hash written in same transaction).
5. Re-evaluate assurance tier: if `last_mfa_at` is older than 12 hours → downgrade to T1 (standard MFA re-challenge required for T2 actions).
6. Return new access token.

### Logout flow

1. Client presents valid access token.
2. auth-service marks `session.status = revoked`.
3. Refresh token hash is zeroed; cannot be reused.
4. Write auth_event (event_type=session_revoked).
5. Emit to audit.append (DP-036).
