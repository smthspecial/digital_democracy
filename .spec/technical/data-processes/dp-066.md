## Description

Responds to suspicious authentication events detected by auth-service. Suspends the affected session, notifies the citizen, and requires re-authentication with T3 assurance to restore access.

### Anomaly conditions

| Condition | Detection |
|-----------|-----------|
| `new_device` | Device fingerprint not seen in last 90 days for this citizen |
| `new_country` | Login from a country not associated with this citizen's last 10 sessions |
| `concurrent_geos` | Two active sessions with IPs in geographically incompatible regions (distance > 500km, time delta < 2h) |
| `mfa_brute_force` | 5 or more MFA failures within 10 minutes |
| `token_reuse` | A refresh token hash is presented that was already rotated (potential token theft) |

### Response flow

1. auth-service emits anomaly event to identity.check queue with `citizen_id`, `session_id`, and `anomaly_reason`.
2. SRV-017 worker immediately sets `session.status = suspended` for the affected session (TBL-037).
3. Any access token issued by that session is rejected on next use (services validate session status on token check).
4. Write auth_event row (TBL-039, event_type=anomaly_detected).
5. Emit to audit.append (DP-036).
6. Dispatch notification to citizen via DP-039: anomaly type, time, approximate location — with instructions to re-authenticate if the login was legitimate, or to contact support if not.
7. To restore access: citizen must complete a fresh T3 step-up (DP-061, facial or passkey). On success: `session.status = active`.
8. If the citizen reports the session as unauthorized: `session.status = revoked` permanently. identity-service is notified to flag the citizen's account for manual review (SRV-001).
