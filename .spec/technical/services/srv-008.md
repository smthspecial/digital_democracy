## Responsibility

Manages the full voting lifecycle: scheduling sessions, enforcing cooling-off periods, issuing blind eligibility tokens, accepting encrypted ballots, computing tallies, and certifying outcomes. Cryptographically separates voter identity from ballot content so eligibility is provable but choices are anonymous (ADR-002). Supports multiple voting methods (ranked-choice, approval, preference-score, comparative) and decision-type threshold rules.

## Key rules

- **Cryptographic separation**: `eligibility_token` has `citizen_id`; `ballot` does NOT. The bridge is a blind-signed one-time token that cannot be rejoined to a citizen record (ADR-002, FR-003, NFR-001).
- Token issuance (DP-025) reads all eligible citizens (via jurisdiction-service and residency checks) and inserts one `eligibility_token` per citizen. Unique constraint on `(vote_session_id, citizen_id)` makes it idempotent.
- Casting a ballot (DP-016): (1) verify token is valid and `used=false`, (2) write `ballot` row (no citizen_id), (3) mark `eligibility_token.used=true`. These three steps are a single database transaction.
- `ballot.encrypted_choice` uses threshold cryptography; decryption keys are distributed across independent key holders. No single operator can decrypt.
- `ballot.verification_code` lets the voter confirm inclusion without proving their choice to a third party (coercion resistance, FR-004).
- Tally (DP-026) holds a distributed lock per session; exactly-once semantics enforced via session-level idempotency key.
- Constitutional review (DP-034) must be `cleared` before a session can be scheduled.
- `min_participation` quorum is enforced at certification (DP-027): failure sets status to `closed` (failed quorum), not `certified`.
