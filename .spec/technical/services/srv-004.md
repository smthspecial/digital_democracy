## Responsibility

Manages the full lifecycle of proposals from draft through archived. Enforces all lifecycle gates: support threshold before development (FR-017), scope assignment before voting (FR-008), and complete funding information before voting (FR-037). Routes proposals through scope assignment and constitutional review. Coordinates competing proposals on the same problem (FR-018).

| Table | Role |

| TBL-009 proposal_constraint | Primary owner — agreed constraints preceding solution design |
| TBL-010 proposal_budget | Primary owner — cost, funding source, maintenance and long-term costs |

## Key rules

- A proposal cannot advance from `development` to `voting` unless `proposal_budget` is complete (non-null `cost`, `funding_source`, `maintenance_cost`, `expected_benefits`) and `scope_jurisdiction_id` is set (FR-008, FR-037). This gate is checked by the DP-029 worker.
- Multiple proposals can share the same `problem_id` (FR-018); they are presented as competing options during deliberation and voting.
- `support_threshold` is calculated from the population of the assigned `scope_jurisdiction` and is public and configurable per jurisdiction (FR-017).
- Scope challenges (DP-020) enqueue DP-030 and may pause the voting phase for the affected proposal.
- The `status` state machine: `draft → gathering_support → development → voting → approved | rejected | archived`.
- All status transitions are emitted to `audit-service`.

## Dependencies

- Emits to: `voting-service` (when status=voting, triggers vote session creation), `audit-service` (DP-036), `notification-service` (DP-039), `audit-service` (constitutional review queue via DP-034)
- Read by: `voting-service` (vote options), `deliberation-service` (context), `competency-service` (assessment target)
- Reads from: `problem-service` (threshold events via DP-028), `jurisdiction-service` (scope assignment)
