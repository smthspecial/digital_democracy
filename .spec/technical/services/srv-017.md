## Responsibility

Owns the full authentication lifecycle: session issuance and rotation, MFA factor enrollment and verification, step-up challenge orchestration, biometric matching, and anomaly detection. Is the sole issuer of access tokens. Every other service validates access tokens against auth-service without implementing its own session logic.

auth-service does not make authorization decisions — it only asserts identity and assurance tier. Authorization (which actions a citizen may perform) is determined by each service using AUTH-010 and the actor's roles in TBL-032.

## Owned Tables

| Table | Role |
|-------|------|
| TBL-037 session | Primary owner — all active sessions, refresh tokens, device bindings |
| TBL-038 mfa_factor | Primary owner — enrolled MFA factors per citizen |
| TBL-039 auth_event | Primary owner — all authentication events (login, MFA, anomalies) |

## Operations

| Operation | Mode | Process |
|-----------|------|---------|
| Initiate login (issue session) | sync | DP-059 |
| Enroll MFA factor | sync | DP-060 |
| Complete step-up MFA challenge | sync | DP-061 |
| Revoke session (logout) | sync | DP-059 (logout variant) |
| Validate access token | sync | internal — called by all other services |
| Refresh access token | sync | internal — exchange refresh token for new access token |
| Anomaly detection and session suspension | async | DP-066 |
| Purge expired sessions | cron | DP-067 |

## Key rules

- Access tokens have a 15-minute TTL. Refresh tokens are server-side, rotated on each use, and bound to `(device_fingerprint, ip_subnet)` at the time of issuance.
- A mismatched device fingerprint or IP subnet on refresh triggers DP-066 (anomaly).
- MFA tier is embedded in the access token as `assurance_tier: T1 | T2 | T3`. Other services check this field — they never re-invoke MFA logic themselves.
- Facial biometric matching runs entirely within auth-service. The biometric embedding stored in TBL-038 never leaves this service. Raw capture frames are discarded immediately after matching; only the match result and confidence score are retained in TBL-039.
- Liveness detection is mandatory for all facial matches. A static image or video replay must be rejected.
- At least one MFA factor must be enrolled before any T2 action is permitted. auth-service rejects T2 token requests from citizens with no enrolled factor.
- Facial biometric is required for T3 actions unless a passkey is enrolled and verified. If neither is available, T3 actions are blocked until enrollment.
- A citizen's sessions may be force-revoked by a multi-approval governance decision (DP-035) — for example, as part of an identity suspension (DP-042). auth-service exposes an internal `revoke_all_sessions(citizen_id)` endpoint consumed only by identity-service.
- All auth events (login, MFA success/failure, anomaly, revocation) are emitted to audit.append (DP-036).

## Dependencies

- Reads from: `identity-service` — checks `citizen.status` before issuing any session
- Emits to: `audit-service` (DP-036), `notification-service` (DP-039) on anomaly or forced revocation
- Called by: all services — token validation on every authenticated request
