## Responsibility

Accepts citizen-submitted problems and tracks public endorsement. Surfaces community priorities so that proposal development starts from demonstrated need rather than top-down initiative. Exposes support counts to the proposal-service so that threshold gates can fire.

## Owned Tables

| Table | Role |
|-------|------|
| TBL-006 problem | Primary owner — submission, status, affected area, candidate scope |
| TBL-007 problem_support | Primary owner — per-citizen endorsements; unique per (problem, citizen) |

## Operations

| Operation | Mode | Process |
|-----------|------|---------|
| Submit problem | sync | DP-003 |
| Endorse problem | sync | DP-004 |
| Proposal threshold check | async | DP-028 (triggered by DP-004) |

## Key rules

- Any active citizen may submit a problem (FR-015). Submitted problems are public immediately (FR-016).
- `problem_support` has a unique constraint on `(problem_id, citizen_id)` — one endorsement per citizen per problem.
- When DP-004 writes a new `problem_support` row, it enqueues DP-028. The worker reads the current `support_count` on the associated `proposal` and determines whether any linked proposal has crossed its `support_threshold`.
- Problem status transitions: `open` → `proposing` (when a proposal reaches development) → `closed` (when a proposal is approved or all proposals are rejected/archived).
- Audit event emitted on problem creation and status change.

## Dependencies

- Emits to: `proposal-service` (threshold events via DP-028), `audit-service` (DP-036), `notification-service` (DP-039)
- Read by: `proposal-service` (linking proposals to problems), `deliberation-service` (context)
